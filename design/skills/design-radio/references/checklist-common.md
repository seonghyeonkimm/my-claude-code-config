# Design RADIO 공통 체크리스트

FE/BE 무관하게 적용되는 공통 질문. 트랙별 상세는 `checklist-fe.md` 또는 `checklist-be.md` 참조.

---

## C — Context (Gate In)

- [ ] 관련 기존 코드를 탐색했는가?
- [ ] 재사용 가능한 모듈/함수/컴포넌트를 식별했는가?
- [ ] 프로젝트의 기존 패턴과 컨벤션을 파악했는가?
- [ ] 변경이 필요한 기존 코드의 범위를 파악했는가?

### 산출물

```markdown
## Context
- 재사용: {기존 모듈/함수 목록 + 파일 경로}
- 컨벤션: {프로젝트 패턴 요약}
- 변경 대상: {수정할 기존 코드 목록}
```

---

## R — Requirements

- [ ] **한 문장 정의**: "X를 하면 Y가 된다" 형식으로 적을 수 있는가?
- [ ] **기능 요구사항**: 핵심 유저 플로우를 나열했는가?
- [ ] **비기능 요구사항**: 성능, 보안, 접근성 중 해당 항목을 식별했는가?
- [ ] **스코프 아웃**: "이번에 하지 않는 것"을 명시했는가?

### 산출물

```markdown
## Requirements
- 문제: {한 문장}
- 기능: {핵심 유저 플로우 목록}
- 비기능: {해당 NFR 항목}
- 스코프 아웃: {하지 않는 것}
```

---

## A — Architecture

- [ ] 주요 컴포넌트를 식별했는가? (3~7개)
- [ ] 각 컴포넌트의 단일 책임을 정의했는가?
- [ ] 컴포넌트 간 통신 방식을 정했는가?
- [ ] 계산/로직이 수행되는 위치를 정했는가?

### 산출물

```
[컴포넌트 A] --호출--> [컴포넌트 B]
     |                      |
     v                      v
[컴포넌트 C]          [컴포넌트 D]
```

+ 각 컴포넌트의 한 줄 책임 설명

---

## D — Data Model

- [ ] 핵심 엔티티를 식별했는가?
- [ ] 각 엔티티의 필드, 타입, 제약조건을 정의했는가?
- [ ] 엔티티 간 관계를 정의했는가? (1:1, 1:N, N:M)
- [ ] 데이터의 출처와 생명주기를 분류했는가?

---

## I — Interface Definition

- [ ] 각 컴포넌트 경계의 Input/Output을 정의했는가?
- [ ] 에러 처리 계약을 정의했는가?
- [ ] 타입을 명시했는가?

---

## O — Optimizations

R 단계의 NFR에서 직접 도출된 항목만 기록한다.
"해당 없음"이 대부분이어도 정상. 추측성 최적화는 하지 않는다.

---

## V — Verification (Gate Out)

Given/When/Then 형식으로 **핵심 행위 3~5개**를 작성한다.
이것이 "완료의 정의"이며, 순서는 정상 → 에러 → 엣지.

```markdown
| # | Given | When | Then |
|---|-------|------|------|
| 1 | {초기 상태} | {행동} | {기대 결과} |
```

- [ ] 정상 케이스가 1개 이상 있는가?
- [ ] 에러 케이스가 1개 이상 있는가?
- [ ] 각 TC가 고유한 경계 조건을 검증하는가? (중복 TC 제거)
- [ ] 이 TC를 통과하면 "완료"라고 말할 수 있는가?
