# Think — Self Consistency 분석

주어진 과제에 대해 3개의 독립적인 관점에서 접근 방안을 생성하고, 교차 비교하여 가장 견고한 방안을 추천한다.

## 원리

Self Consistency는 동일 문제에 대해 독립적인 추론 경로를 여러 개 생성한 뒤 결과를 비교하는 기법이다. 각 경로가 서로 다른 우선순위와 가정으로 출발하므로 앵커링 편향을 줄이고, 수렴하는 결론은 신뢰도가 높다. 여기에 웹 검색을 통한 외부 검증 단계를 추가하여, 추론 결과를 실세계 정보(공식 문서, 벤치마크, 모범 사례)와 대조함으로써 환각(hallucination)과 오래된 정보에 기반한 판단을 방지한다.

## Input

- `$ARGUMENTS.task`: 분석할 작업 설명

## Execution Flow

### Phase 1: 컨텍스트 수집

1. `$ARGUMENTS.task`에서 핵심 키워드와 의도를 추출한다.

2. 과제가 코드 관련인 경우:
   - Grep으로 키워드 관련 파일 탐색
   - Glob으로 관련 디렉토리 구조 파악
   - 핵심 파일 2-3개를 Read하여 현재 구현 상태 파악

3. 수집한 정보를 아래 형식으로 정리한다. 이 요약을 Phase 2에서 3개 agent에 동일하게 전달한다.

```
## 컨텍스트 요약
- 과제: {task 설명}
- 관련 파일: {파일 목록}
- 현재 상태: {현재 구현/상황 요약}
- 제약 조건: {식별된 제약사항}
```

코드와 무관한 일반 과제(설계 결정, 전략 질문 등)인 경우 코드베이스 탐색을 건너뛰고 과제 설명만으로 컨텍스트를 구성한다.

### Phase 2: 독립 분석 (3 병렬 Task)

3개의 Task를 **반드시 동시에** 생성한다. 각 agent는 동일한 컨텍스트를 받되 서로 다른 관점에서 분석한다.

> 핵심: 각 Task의 prompt에 다른 관점의 존재를 언급하지 않는다. 앵커링 방지를 위함이다.

```
  - 리스크 최소화: 부작용과 회귀 가능성을 줄이는 방향
```

### Phase 3: 웹 검증 (Web Search Verification)

3개 agent의 분석 결과에서 검증이 필요한 핵심 주장을 추출하고, 웹 검색으로 실세계 정보와 대조한다.

#### 3-1: 검증 대상 추출

3개 분석 결과를 스캔하여 아래 범주의 주장을 추출한다:

| 범주 | 예시 | 추출 기준 |
|------|------|-----------|
| 사실적 주장 | "React 18부터 useSyncExternalStore가 권장됨" | 특정 버전, API, 도구에 대한 사실 진술 |
| 모범 사례 | "상태 관리는 서버/클라이언트 상태를 분리하는 것이 표준" | 업계 표준/패턴에 대한 주장 |
| 기술적 추천 | "Zustand가 이 규모에 적합하다" | 특정 도구/라이브러리/접근법 추천 |
| 성능/규모 주장 | "이 접근은 1000개 아이템까지 문제없다" | 수치적 주장 또는 규모 관련 판단 |
| 리스크 판단 | "마이그레이션 시 breaking change 위험이 낮다" | 리스크/안전성에 대한 평가 |

추출 결과를 아래 형식으로 정리한다:

```
## 검증 대상 목록
| # | 주장 | 출처 관점 | 범주 | 검증 쿼리 |
|---|------|-----------|------|-----------|
| 1 | {주장 내용} | A/B/C | 사실적 주장 | {WebSearch 쿼리} |
| 2 | {주장 내용} | A/B/C | 모범 사례 | {WebSearch 쿼리} |
```

> 최대 5개까지만 추출한다. 3개 관점에서 공통으로 등장하는 주장은 이미 수렴 신호이므로 우선순위를 낮추고, **관점 간 충돌하는 주장을 우선 검증**한다.

#### 3-2: 웹 검색 실행

각 검증 대상에 대해 `WebSearch`를 실행한다.

**쿼리 전략:**
- 사실적 주장: `{기술명} {버전} official documentation {핵심 키워드}`
- 모범 사례: `{기술명} best practices {연도}`
- 기술적 추천: `{도구명} vs {대안} comparison {연도}`
- 성능 주장: `{기술명} performance benchmark {규모 키워드}`
- 리스크 판단: `{기술명} migration guide breaking changes`

필요 시 `WebFetch`로 검색 결과 중 공식 문서나 신뢰도 높은 소스의 상세 내용을 확인한다.

**소스 신뢰도 기준:**
- 높음: 공식 문서, RFC, 공식 블로그
- 중간: 유명 기술 블로그 (engineering blog), Stack Overflow 고득표 답변, 벤치마크 리포지토리
- 낮음: 개인 블로그, 오래된(2년 이상) 자료

#### 3-3: 검증 결과 정리

```
## 웹 검증 결과

| # | 주장 | 검증 결과 | 신뢰도 | 근거 |
|---|------|-----------|--------|------|
| 1 | {주장} | ✅ 확인됨 / ⚠️ 부분 일치 / ❌ 반박됨 / ➖ 검증 불가 | 높음/중간/낮음 | {출처 요약} |

### 주요 발견 사항
- {검증 과정에서 발견된 중요한 추가 정보}
- {분석에서 누락된 중요 맥락}
```

### Phase 4: 교차 비교 & 합성

3개 agent의 결과를 수집한 뒤:

1. **수렴 분석**: 3개 접근에서 공통으로 등장하는 요소를 식별한다.
   - 3/3 수렴 → 높은 신뢰도 신호
   - 2/3 수렴 → 주류 의견 + 소수 의견
   - 0/3 수렴 → 과제의 복잡성/다면성 신호

2. **비교표 출력**:

```
## 접근 비교

| 기준 | A. Pragmatist | B. Architect | C. Contrarian |
|------|--------------|-------------|---------------|
| 핵심 아이디어 | {1줄 요약} | {1줄 요약} | {1줄 요약} |
| 작업량 | {규모} | {규모} | {규모} |
| 최대 강점 | {강점} | {강점} | {강점} |
| 최대 약점 | {약점} | {약점} | {약점} |
| 리스크 | {리스크} | {리스크} | {리스크} |
| 웹 검증 | {검증 요약} | {검증 요약} | {검증 요약} |
```

3. **수렴 지점 정리**:

```
## 수렴 분석

### 공통 요소 (신뢰도 높음)
- {3개 접근 모두에서 등장한 요소}

### 분기점 (핵심 결정 사항)
- {접근들이 갈라지는 지점과 그 이유}
```

4. **웹 검증 반영**:

```
## 웹 검증 반영

### 검증으로 확인된 사항
- {웹 검색이 뒷받침하는 분석 요소}

### 검증으로 수정이 필요한 사항
- {웹 검색 결과와 충돌하는 분석 요소 및 수정 방향}

### 추가 발견 사항
- {검증 과정에서 새로 확인된 맥락 — 분석에 없던 정보}
```

5. **추천 합성**:

```
## 추천

**추천 접근: {A/B/C 또는 하이브리드}**

{추천 이유. 수렴 분석 결과와 웹 검증 결과를 근거로 활용.}
{웹 검증에서 확인된/수정된 사항을 반영한 근거 보강.}
{하이브리드인 경우 어떤 요소를 조합하는지 구체적으로 설명.}
```

### Phase 5: 사용자 선택

AskUserQuestion으로 선택지를 제시한다:
- **A** — Pragmatist 접근으로 진행
- **B** — Architect 접근으로 진행
- **C** — Contrarian 접근으로 진행
- **R** — 추천 접근(하이브리드)으로 진행
- **추가 분석** — 특정 관점이나 기준을 추가하여 재분석

선택 후:
- A/B/C/R 선택 시 → 선택한 접근의 실행 단계를 정리하여 출력 후 종료
- 추가 분석 시 → 피드백을 반영하여 Phase 2부터 재실행

## Error Handling

| 상황 | 대응 |
|------|------|
| task 인자 없음 | "어떤 과제를 분석할까요?" 질문 |
| 3개 접근이 실질적으로 동일 | "3개 관점이 동일한 결론에 수렴했습니다 — 신뢰도가 높습니다." 출력 후 세부 차이만 비교 |
| 코드베이스 탐색 실패 | 코드 컨텍스트 없이 일반 분석으로 진행 (경고 출력) |
| Task 하나가 형식에 맞지 않는 결과 반환 | 해당 관점을 "분석 실패"로 표시하고 나머지 2개로 비교 진행 |
| WebSearch 결과 부족 | 해당 주장을 "검증 불가"로 표시하고, 검증 없이 Phase 4 비교 진행. 검증 불가 사항은 비교표에 명시 |
| WebSearch 전체 실패 (네트워크 등) | Phase 3을 건너뛰고 기존 방식으로 Phase 4 진행. "⚠️ 웹 검증을 수행하지 못했습니다" 경고 출력 |
| WebFetch 접근 불가 | WebSearch 결과 스니펫만으로 판단. 상세 확인 불가 시 신뢰도를 "낮음"으로 표시 |

## Example

```
사용자: /think 장바구니 페이지의 상태 관리를 어떤 방식으로 구현할지

Claude: 컨텍스트를 수집합니다...
  → src/pages/cart/ 관련 파일 5개 발견
  → 현재 React Query + useState 혼용 상태

[3개 관점 병렬 분석 실행]

[웹 검증] 핵심 주장 검증 중...
  → "Zustand가 이 규모에 적합" — ✅ 공식 문서에서 중소규모 앱에 권장 확인
  → "URL State만으로 장바구니 구성 가능" — ⚠️ 복잡한 장바구니에는 제한적이라는 사례 다수
  → "React Query는 서버 상태 단일 소스로 적합" — ✅ TanStack Query 공식 문서 확인

## 접근 비교

| 기준 | A. Pragmatist | B. Architect | C. Contrarian |
|------|--------------|-------------|---------------|
| 핵심 아이디어 | useState 범위만 정리 | Zustand로 상태 분리 | URL + Server State만으로 구성 |
| 작업량 | 소 (2-3 파일) | 중 (5-8 파일) | 중 (4-6 파일) |
| 최대 강점 | 리스크 제로 | 명확한 관심사 분리 | 근본적 단순화 |
| 최대 약점 | 규모 한계 | 새 의존성 | 임시 상태 처리 어려움 |

## 수렴 분석
### 공통 요소
- React Query를 서버 상태의 단일 소스로 유지 (3/3)
```
