# Think — Self Consistency 분석

주어진 과제에 대해 3개의 독립적인 관점에서 접근 방안을 생성하고, 교차 비교하여 가장 견고한 방안을 추천한다.

## 원리

Self Consistency는 동일 문제에 대해 독립적인 추론 경로를 여러 개 생성한 뒤 결과를 비교하는 기법이다. 각 경로가 서로 다른 우선순위와 가정으로 출발하므로 앵커링 편향을 줄이고, 수렴하는 결론은 신뢰도가 높다.

## Input

- `$ARGUMENTS.task`: 분석할 작업 설명

## Execution Flow

### Phase 1: 컨텍스트 수집

1. `$ARGUMENTS.task`에서 핵심 키워드와 의도를 추출한다.

2. 과제가 코드 관련인 경우:
   - Grep으로 키워드 관련 파일 탐색
   - Glob으로 관련 디렉토리 구조 파악
   - 핵심 파일 2-3개를 Read하여 현재 구현 상태 파악

3. 수집한 정보를 아래 형식으로 정리한다. 이 요약을 Phase 2에서 3개 agent에 동일하게 전달한다.

```
## 컨텍스트 요약
- 과제: {task 설명}
- 관련 파일: {파일 목록}
- 현재 상태: {현재 구현/상황 요약}
- 제약 조건: {식별된 제약사항}
```

코드와 무관한 일반 과제(설계 결정, 전략 질문 등)인 경우 코드베이스 탐색을 건너뛰고 과제 설명만으로 컨텍스트를 구성한다.

### Phase 2: 독립 분석 (3 병렬 Task)

3개의 Task를 **반드시 동시에** 생성한다. 각 agent는 동일한 컨텍스트를 받되 서로 다른 관점에서 분석한다.

> 핵심: 각 Task의 prompt에 다른 관점의 존재를 언급하지 않는다. 앵커링 방지를 위함이다.

#### Task A: Pragmatist (실용주의)

```
Task(subagent_type: "general-purpose", prompt: "
  당신은 실용주의 엔지니어입니다. 다음 과제에 대한 접근 방안을 제시하세요.

  ## 당신의 우선순위
  - 최소 변경: 기존 코드/구조를 최대한 활용
  - 빠른 전달: 가장 짧은 경로로 목표 달성
  - 검증된 방법: 팀이 이미 익숙한 패턴과 도구 선호
  - 리스크 최소화: 부작용과 회귀 가능성을 줄이는 방향

  ## 과제
  {$ARGUMENTS.task}

  ## 컨텍스트
  {Phase 1에서 수집한 컨텍스트 요약}

  ## 출력 형식 (반드시 준수)
  ### 접근 방안
  {구체적인 접근 방법을 단계별로 설명}

  ### 핵심 근거
  {이 접근을 선택한 이유 2-3가지}

  ### 트레이드오프
  - 장점: {이 접근의 강점}
  - 단점: {이 접근의 약점 또는 포기하는 것}
  - 리스크: {잠재적 위험 요소}

  ### 예상 작업량
  {대략적인 작업 규모와 영향 범위}
")
```

### Phase 3: 교차 비교 & 합성

3개 agent의 결과를 수집한 뒤:

1. **수렴 분석**: 3개 접근에서 공통으로 등장하는 요소를 식별한다.
   - 3/3 수렴 → 높은 신뢰도 신호
   - 2/3 수렴 → 주류 의견 + 소수 의견
   - 0/3 수렴 → 과제의 복잡성/다면성 신호

2. **비교표 출력**:

```
## 접근 비교

| 기준 | A. Pragmatist | B. Architect | C. Contrarian |
|------|--------------|-------------|---------------|
| 핵심 아이디어 | {1줄 요약} | {1줄 요약} | {1줄 요약} |
| 작업량 | {규모} | {규모} | {규모} |
| 최대 강점 | {강점} | {강점} | {강점} |
| 최대 약점 | {약점} | {약점} | {약점} |
| 리스크 | {리스크} | {리스크} | {리스크} |
```

3. **수렴 지점 정리**:

```
## 수렴 분석

### 공통 요소 (신뢰도 높음)
- {3개 접근 모두에서 등장한 요소}

### 분기점 (핵심 결정 사항)
- {접근들이 갈라지는 지점과 그 이유}
```

4. **추천 합성**:

```
## 추천

**추천 접근: {A/B/C 또는 하이브리드}**

{추천 이유. 수렴 분석 결과를 근거로 활용.}
{하이브리드인 경우 어떤 요소를 조합하는지 구체적으로 설명.}
```

### Phase 4: 사용자 선택

AskUserQuestion으로 선택지를 제시한다:
- **A** — Pragmatist 접근으로 진행
- **B** — Architect 접근으로 진행
- **C** — Contrarian 접근으로 진행
- **R** — 추천 접근(하이브리드)으로 진행
- **추가 분석** — 특정 관점이나 기준을 추가하여 재분석

선택 후:
- A/B/C/R 선택 시 → 선택한 접근의 실행 단계를 정리하여 출력 후 종료
- 추가 분석 시 → 피드백을 반영하여 Phase 2부터 재실행

## Error Handling

| 상황 | 대응 |
|------|------|
| task 인자 없음 | "어떤 과제를 분석할까요?" 질문 |
| 3개 접근이 실질적으로 동일 | "3개 관점이 동일한 결론에 수렴했습니다 — 신뢰도가 높습니다." 출력 후 세부 차이만 비교 |
| 코드베이스 탐색 실패 | 코드 컨텍스트 없이 일반 분석으로 진행 (경고 출력) |
| Task 하나가 형식에 맞지 않는 결과 반환 | 해당 관점을 "분석 실패"로 표시하고 나머지 2개로 비교 진행 |

## Example

```
사용자: /think 장바구니 페이지의 상태 관리를 어떤 방식으로 구현할지

Claude: 컨텍스트를 수집합니다...
  → src/pages/cart/ 관련 파일 5개 발견
  → 현재 React Query + useState 혼용 상태

[3개 관점 병렬 분석 실행]

## 접근 비교

| 기준 | A. Pragmatist | B. Architect | C. Contrarian |
|------|--------------|-------------|---------------|
| 핵심 아이디어 | useState 범위만 정리 | Zustand로 상태 분리 | URL + Server State만으로 구성 |
| 작업량 | 소 (2-3 파일) | 중 (5-8 파일) | 중 (4-6 파일) |
| 최대 강점 | 리스크 제로 | 명확한 관심사 분리 | 근본적 단순화 |
| 최대 약점 | 규모 한계 | 새 의존성 | 임시 상태 처리 어려움 |

## 수렴 분석
### 공통 요소
- React Query를 서버 상태의 단일 소스로 유지 (3/3)

## 추천
**추천 접근: A + B 하이브리드**
현재 패턴 유지하되, 여러 컴포넌트가 공유하는 클라이언트 상태만 Zustand로 분리.

[사용자 선택: A / B / C / R / 추가 분석]
```
